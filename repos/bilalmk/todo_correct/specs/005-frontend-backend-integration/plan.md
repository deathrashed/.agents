## 🔄 Critical Update: Better Auth UUID Integration (2026-01-02)

**Issue**: Dual authentication system caused failures - custom `users` table conflicted with Better Auth `user` table.

**Solution Implemented**: Hybrid ID approach using Better Auth as single source of truth
- Extended Better Auth `user` table with UUID column
- All foreign keys updated to point to `user.uuid` (not `users.id`)
- JWT tokens include custom `uuid` claim
- UUID auto-generated by database `server_default=gen_random_uuid()`
- Custom `users` table dropped

**Migrations Applied**: 3 new migrations (see `data-model.md` and `BETTER_AUTH_UUID_INTEGRATION.md`)

**Impact on Implementation**:
- ✅ User model now maps to Better Auth `user` table (singular, not `users`)
- ✅ Backend extracts `uuid` from JWT custom claim (not `sub`)
- ✅ All queries use `User.uuid` for user identification
- ❌ Backend auth endpoints removed (Better Auth handles registration/login)
- ✅ Foreign keys: `tasks.user_id` → `user.uuid`, `tags.user_id` → `user.uuid`

**For Future Spec Regeneration**: Always use Better Auth `user` table with UUID extension. See `BETTER_AUTH_UUID_INTEGRATION.md` for complete guide.

## Summary

Integrate the Next.js frontend (Spec 004) with FastAPI backend (Spec 003) using Better Auth JWT authentication via JWKS verification. Replace custom JWT validation in backend with EdDSA/Ed25519 signature verification using Better Auth's public keys. Replace all mock data in frontend contexts (TaskContext, TagContext) with real API calls to backend endpoints. Implement secure cookie-based JWT transmission with CSRF protection, user isolation enforcement, and comprehensive error handling with structured logging and correlation IDs for end-to-end request tracing.

**Updated**: Better Auth serves as single source of truth for user management with hybrid ID approach (String ID for Better Auth + UUID for application consistency).

## Technical Context

**Language/Version**: Python 3.11+ (backend), TypeScript/Next.js 16+ (frontend)
**Primary Dependencies**:
- Backend: FastAPI, python-jose[cryptography] (better EdDSA/Ed25519 support, actively maintained, aligned with betterauth-fastapi-jwt-bridge skill), httpx (JWKS fetch), asyncio, SQLModel, Neon PostgreSQL
- Frontend: Next.js 16+, Better Auth SDK, TypeScript, React Query (optional), shadcn/ui, Tailwind CSS
- Auth: Better Auth JWKS verification (EdDSA/Ed25519)

**Storage**: Neon Serverless PostgreSQL (existing from Spec 002)
**Testing**:
- Backend: pytest, pytest-asyncio
- Frontend: Vitest, React Testing Library, MSW (Mock Service Worker)
- E2E: Playwright or Cypress

**Target Platform**: Linux server (backend), Browser (frontend), WSL 2 (development)
**Project Type**: Web application (frontend + backend monorepo)
**Performance Goals**:
- JWT verification: <50ms per request (p95)
- JWKS cache hit rate: >95%
- API calls with auth: <500ms p95
- JWKS fetch (cache miss): <100ms
- JWKS fetch (cache hit): <1ms

**Constraints**:
- JWKS cache TTL: 1 hour maximum
- Retry logic: 3 attempts with exponential backoff
- Frontend timeout: 30 seconds for API calls
- Cookie-based JWT transmission only (no Authorization header)
- CORS configuration: specific origin, credentials: true
- Structured logging overhead: <5ms p95 latency

**Scale/Scope**:
- Support 100 concurrent users per instance
- Test coverage: ≥80% overall, ≥90% auth modules
- 14 existing API endpoints (from Spec 003)
- 15+ UI components (from Spec 004)
- Multi-user support with user isolation
