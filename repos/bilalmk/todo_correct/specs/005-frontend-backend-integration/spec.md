### Session 2026-01-01

- Q: JWT token storage method (httpOnly cookie vs localStorage vs sessionStorage)? → A: httpOnly cookie - Better Auth manages cookie automatically, immune to XSS token theft, requires CSRF protection
- Q: JWT transmission method (Cookie header vs Authorization header vs both)? → A: Cookie header (automatic) - Backend reads JWT from Cookie header sent automatically by browser, requires CORS credentials:true, simpler frontend
- Q: CSRF protection mechanism (SameSite attribute vs Better Auth built-in CSRF vs Double-submit cookie)? → A: Better Auth built-in CSRF tokens - Better Auth SDK handles token generation/validation automatically, proper protection without UX degradation
- Q: How does frontend obtain user_id for API URLs (Better Auth session API vs Extract from JWT vs /me endpoint)? → A: Better Auth session API - Frontend calls auth.getSession() to get user object with user_id, standardized approach, type-safe
- Q: Pagination approach for large task lists (Infinite scroll vs Traditional pagination vs Hybrid Load More)? → A: Traditional pagination - Page numbers with Previous/Next buttons, easier to implement, allows jumping to specific pages, better for finding specific tasks
- Q: Error logging and monitoring strategy? → A: Structured logging (JSON) with correlation IDs to console/file (external monitoring services like Sentry/DataDog deferred to Phase V) - Enables request tracing and debugging without performance impact

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Secure User Registration and Authentication (Priority: P1)

As a new user, I want to register and log in via Better Auth so I can access my personal todo application with secure authentication that doesn't require manual JWT handling.

**Why this priority**: This is the foundational capability that gates access to all other features. Without authentication, users cannot access the application or their data. This establishes the security boundary for the entire system.

**Independent Test**: Can be fully tested by completing registration flow (email, password, name), receiving JWT token, and successfully accessing the dashboard. Delivers immediate value by allowing users to create accounts and log in securely.

**Acceptance Scenarios**:

1. **Given** I am on the registration page, **When** I provide valid email, password (min 8 chars), and name, **Then** Better Auth creates my account, issues a JWT token, and redirects me to the dashboard
2. **Given** I have registered an account, **When** I log in with correct credentials, **Then** I receive a valid JWT token and can access protected routes
3. **Given** I am logged in, **When** I navigate to protected routes, **Then** my JWT token is automatically included in API requests via Cookie header
4. **Given** I provide invalid credentials, **When** I attempt to log in, **Then** I see a clear error message and remain on the login page

---

### User Story 2 - Complete Task Management with Real Data (Priority: P2)

As an authenticated user, I want to create, update, view, and delete tasks through the UI with all changes persisted to the database, so my task list reflects real data instead of mock data.

**Why this priority**: This delivers the core value proposition of the application - task management. While authentication (P1) is required first, this is what users came to the app to do. Without this, the app has no practical utility.

**Independent Test**: Can be tested by logging in, performing CRUD operations (create task "Buy groceries", update title, mark complete, delete), and verifying persistence across page refreshes and browser sessions.

**Acceptance Scenarios**:

1. **Given** I am logged in, **When** I create a new task with title "Buy groceries" and description "Milk, eggs, bread", **Then** the frontend calls POST /api/v1/{user_id}/tasks and the task appears in my list immediately
2. **Given** I have existing tasks, **When** I update a task's title or description, **Then** the frontend calls PATCH /api/v1/{user_id}/tasks/{id} and changes persist to database
3. **Given** I have tasks in my list, **When** I mark a task as complete, **Then** the frontend calls PATCH /api/v1/{user_id}/tasks/{id}/complete and the task moves to completed state
4. **Given** I have tasks in my list, **When** I delete a task, **Then** the frontend calls DELETE /api/v1/{user_id}/tasks/{id} and the task is removed permanently
5. **Given** I have created tasks, **When** I refresh the page or log out and back in, **Then** my tasks are loaded from the database via GET /api/v1/{user_id}/tasks

---

### User Story 3 - Advanced Filtering and Search with Real Backend Data (Priority: P3)

As an authenticated user, I want to filter tasks by status, priority, tags, and date ranges, and search by keywords, so I can quickly find relevant tasks from my complete task list.

**Why this priority**: Enhances usability for users with many tasks, but the app is still functional without it. Users can manage tasks effectively with just P1+P2. This becomes valuable as task lists grow.

**Independent Test**: Can be tested by creating 20+ tasks with various priorities, tags, and due dates, then applying filters (e.g., "Show only high priority work tasks due this week") and verifying query parameters are correctly mapped to backend API.

**Acceptance Scenarios**:

1. **Given** I have tasks with different statuses, **When** I filter by "active" status, **Then** the frontend calls GET /api/v1/{user_id}/tasks?status=incomplete and displays only incomplete tasks
2. **Given** I have tasks with different priorities, **When** I filter by "high" priority, **Then** the frontend calls GET /api/v1/{user_id}/tasks?priority=high and displays only high-priority tasks
3. **Given** I have tasks tagged as "Work" and "Personal", **When** I select the "Work" tag filter, **Then** the frontend calls GET /api/v1/{user_id}/tasks?tag=Work and displays only work-related tasks
4. **Given** I have multiple tasks, **When** I search for "groceries", **Then** the frontend calls GET /api/v1/{user_id}/tasks?search=groceries and displays matching tasks
5. **Given** I have tasks with various due dates, **When** I sort by due date ascending, **Then** the frontend calls GET /api/v1/{user_id}/tasks?sort_by=due_date&order=asc and displays tasks in order

### User Story 4 - Tag Management Integration (Priority: P3)

As an authenticated user, I want to create, update, and delete custom tags through the UI with changes persisted to the database, so I can organize tasks using my own categorization system.

**Why this priority**: Supports task organization but is not essential for core task management. Users can manage tasks without tags using priorities and status filters from P3.

**Independent Test**: Can be tested by creating tags ("Work", "Personal", "Urgent"), applying them to tasks, renaming tags, deleting unused tags, and verifying all operations persist across sessions.

**Acceptance Scenarios**:

1. **Given** I am logged in, **When** I create a new tag "Work" with color blue, **Then** the frontend calls POST /api/v1/{user_id}/tags and the tag appears in my tag list
2. **Given** I have existing tags, **When** I update a tag's name from "Work" to "Office", **Then** the frontend calls PUT /api/v1/{user_id}/tags/{id} and all tasks with this tag are updated
3. **Given** I have unused tags, **When** I delete a tag, **Then** the frontend calls DELETE /api/v1/{user_id}/tags/{id} and the tag is removed from the system
4. **Given** I have created tags, **When** I refresh the page, **Then** tags are loaded from database via GET /api/v1/{user_id}/tags

### User Story 5 - Secure User Isolation and Token Validation (Priority: P1)

As the system, I want to validate JWT tokens on every request and enforce user isolation, so users can only access their own data and invalid/expired tokens are rejected.

**Why this priority**: Critical for security and data privacy. Without this, the app would have a major security vulnerability. This must be implemented alongside P1 authentication.

**Independent Test**: Can be tested by creating two user accounts, attempting cross-user access (User A trying to access User B's tasks via manipulated URLs), using expired tokens, using tampered tokens, and verifying all return appropriate error codes.

**Acceptance Scenarios**:

1. **Given** a request with a valid JWT token, **When** the backend receives it, **Then** it fetches Better Auth JWKS, validates the EdDSA/Ed25519 signature, and extracts user_id for authorization
2. **Given** a request with an expired JWT token, **When** the backend validates it, **Then** it returns 401 Unauthorized and the frontend redirects to login page
3. **Given** a request with a tampered JWT signature, **When** the backend validates it via JWKS verification, **Then** it returns 401 Unauthorized
4. **Given** User A has a valid JWT token with user_id="user_a", **When** User A tries to access /api/v1/user_b/tasks, **Then** the backend returns 403 Forbidden because JWT user_id doesn't match URL {user_id}
5. **Given** a request without an Authorization header, **When** the backend receives it on a protected route, **Then** it returns 401 Unauthorized
6. **Given** JWKS endpoint is unreachable, **When** the backend tries to validate a token, **Then** it retries 3 times with exponential backoff before returning 503 Service Unavailable

---

### Edge Cases

- **Expired JWT tokens**: Backend returns 401, frontend redirects to login page and Better Auth clears httpOnly cookie
- **Invalid JWT signatures**: JWKS verification fails, backend returns 401, frontend redirects to login
- **Missing/invalid CSRF tokens**: Backend returns 403 Forbidden on state-changing requests, frontend retries once with fresh CSRF token from Better Auth
- **User_id mismatch**: JWT contains user_id="alice" but URL is /api/v1/bob/tasks → 403 Forbidden
- **JWKS endpoint unreachable**: Retry 3 times with exponential backoff (100ms, 200ms, 400ms), fallback to 503 Service Unavailable
- **Network timeout on API calls**: Frontend shows retry toast with manual retry button, timeout after 30 seconds
- **CORS preflight failures**: Backend rejects request if Origin header not in allowlist, frontend shows generic network error
- **Concurrent tab sessions**: Better Auth manages session state across tabs, token refresh in one tab updates all tabs
- **Rapid-fire API requests**: Backend may return stale JWKS data (up to 1 hour old due to cache), acceptable for performance
- **Empty task lists**: Frontend displays "No tasks yet" empty state, not an error
- **Large task lists (1000+ tasks)**: Backend returns paginated results (default 50 tasks per page), frontend implements traditional pagination with page numbers and Previous/Next buttons
- **Special characters in search queries**: Frontend URL-encodes query parameters before sending to backend
- **Simultaneous updates to same task**: Last write wins, no optimistic locking required for MVP
- **Monitoring service unavailable**: Backend logs to console/file as fallback, request processing continues without blocking on async monitoring integration
- **Correlation ID missing in retry**: Frontend generates new correlation_id if original not available, logs warning about trace discontinuity
- **Log volume spike (>1000 requests/min)**: Structured JSON logging enables log aggregation services to handle high volume without performance degradation
- **Sensitive data in error messages**: Backend sanitizes error responses to prevent JWT tokens or internal details from being exposed in logs or user-facing errors

## Requirements *(mandatory)*

### Functional Requirements

#### Backend Authentication & Authorization

- **FR-001**: Backend MUST replace custom JWT validation in `/backend/src/api/deps.py` with Better Auth JWKS verification using EdDSA/Ed25519 algorithm, reading JWT from Cookie header
- **FR-002**: Backend MUST fetch and cache Better Auth JWKS from `BETTER_AUTH_JWKS_URL` environment variable with 1-hour TTL
- **FR-003**: Backend MUST maintain existing `verify_user_match()` dependency for user isolation (JWT user_id matches URL {user_id})
- **FR-004**: Backend MUST preserve all 14 existing endpoint signatures without breaking changes
- **FR-005**: Backend MUST retry JWKS fetch 3 times with exponential backoff (100ms, 200ms, 400ms) before returning 503 on failure
- **FR-006**: Backend MUST extract user_id claim from validated JWT token and pass it to route handlers
- **FR-007**: Backend MUST return 401 Unauthorized for missing, expired, or invalid JWT tokens
- **FR-007a**: Backend MUST validate Better Auth CSRF tokens on state-changing requests (POST, PUT, PATCH, DELETE) and return 403 Forbidden for invalid/missing CSRF tokens
- **FR-008**: Backend MUST return 403 Forbidden when JWT user_id doesn't match URL path {user_id} parameter

#### Frontend Authentication

- **FR-009**: Frontend MUST configure Better Auth client with JWT plugin, EdDSA algorithm specification, and CSRF protection enabled
- **FR-010**: Frontend MUST implement registration flow using Better Auth SDK (email, password, name fields)
- **FR-011**: Frontend MUST implement login flow using Better Auth SDK with error handling for invalid credentials
- **FR-012**: Frontend MUST use Better Auth's automatic httpOnly cookie management for JWT token storage (provides XSS protection and automatic token refresh)
- **FR-013**: Frontend MUST configure fetch() with credentials: 'include' to automatically send httpOnly cookie and CSRF token with all API requests to backend

#### Frontend API Integration

- **FR-014**: Frontend MUST create API client module at `lib/api-client.ts` with credentials: 'include' for automatic cookie transmission
- **FR-014a**: Frontend MUST use Better Auth `auth.getSession()` to obtain authenticated user's user_id for constructing API URLs (e.g., `/api/v1/{user_id}/tasks`)
- **FR-015**: Frontend MUST replace all mock data in TaskContext with real API calls to backend endpoints
- **FR-016**: Frontend MUST replace all mock data in TagContext with real API calls to backend endpoints
- **FR-017**: Frontend MUST map filter UI state to backend query parameters:
  - Status: "active" → `?status=incomplete`, "completed" → `?status=complete`, "all" → omit
  - Priority: "low"/"medium"/"high" → `?priority=...`, "all" → omit
  - Tags: ["Work", "Personal"] → `?tag=Work&tag=Personal`
  - Search: "groceries" → `?search=groceries`
  - Sort: sortBy="due_date", order="asc" → `?sort_by=due_date&order=asc`
  - Pagination: page=2, limit=50 → `?page=2&limit=50`
- **FR-017a**: Frontend MUST implement traditional pagination UI with page numbers, Previous/Next buttons, and page size selector (default 50 tasks per page)

#### Error Handling

- **FR-018**: Frontend MUST handle 401 errors by calling Better Auth signOut() method (clears httpOnly cookie) and redirecting to login page
- **FR-019**: Frontend MUST handle 403 errors by showing toast notification "You don't have permission to access this resource"
- **FR-020**: Frontend MUST handle 404 errors by showing toast notification "Resource not found"
- **FR-021**: Frontend MUST handle 422 errors by showing validation error messages from backend response
- **FR-022**: Frontend MUST handle 500/503 errors by showing toast notification "Server error, please try again later"
- **FR-023**: Frontend MUST handle network timeout errors (30s timeout) by showing retry toast with manual retry button

#### CORS Configuration

- **FR-024**: Backend MUST configure CORS middleware with frontend domain in allowlist (specific origin, not wildcard)
- **FR-025**: Backend MUST set Access-Control-Allow-Credentials: true in CORS configuration to allow httpOnly cookie transmission
- **FR-026**: Backend MUST respond to OPTIONS preflight requests with appropriate Access-Control-Allow-* headers

#### Feature Preservation

- **FR-027**: Integration MUST preserve all existing UI features: filtering, sorting, search, tag management, task CRUD, priorities, due dates, reminders, recurrence patterns
- **FR-028**: Frontend MUST maintain existing UI/UX design from Spec 004 (shadcn/ui components, responsive layout)

#### Logging and Monitoring

- **FR-029**: Backend MUST implement structured logging in JSON format with the following required fields: timestamp, level (DEBUG/INFO/WARN/ERROR), correlation_id, user_id (if available), endpoint, http_method, status_code, duration_ms, error_message (if applicable)
- **FR-030**: Backend MUST generate unique correlation_id (UUID v4) for each incoming request and include it in all log entries and error responses (in response headers as X-Correlation-ID)
- **FR-031**: Frontend MUST extract correlation_id from error response headers and display it in error messages for user support troubleshooting
- **FR-032**: Backend MUST log all authentication events at INFO level: successful logins (user_id, timestamp), failed logins (email, reason, timestamp), JWT validation failures (reason, correlation_id), JWKS fetch operations (success/failure, cache hit/miss)
- **FR-033**: Backend MUST log all authorization failures at WARN level: user_id mismatches (requested_user_id, jwt_user_id, correlation_id), CSRF validation failures
- **FR-034**: Backend MUST implement structured console/file logging for error tracking and performance monitoring without blocking request processing (external monitoring services like Sentry, New Relic, or DataDog deferred to Phase V)
- **FR-035**: Backend MUST NOT log sensitive data: full JWT tokens, passwords, CSRF tokens, or complete JWKS responses (only log metadata like cache status, validation result)
- **FR-036**: Frontend MUST send correlation_id in custom header (X-Correlation-ID) if retrying a failed request to maintain trace continuity

### Key Entities

- **JWT Token**: Contains claims (user_id, email, exp, iat), signed with EdDSA/Ed25519 algorithm, issued by Better Auth, stored in httpOnly cookie, validated by backend via JWKS
- **CSRF Token**: Generated by Better Auth, automatically included in state-changing requests, validated by backend to prevent cross-site request forgery attacks
- **JWKS (JSON Web Key Set)**: Public keys for JWT verification, fetched from Better Auth JWKS endpoint, cached with 1-hour TTL, used to verify EdDSA signatures
- **API Client**: TypeScript module that wraps fetch() with credentials: 'include' for automatic cookie and CSRF token transmission, error handling, and response parsing
- **User Session**: Managed by Better Auth, includes JWT token storage in httpOnly cookie, automatic token refresh, CSRF protection, and multi-tab synchronization
- **Correlation ID**: UUID v4 generated per request for end-to-end tracing, included in all log entries and error responses via X-Correlation-ID header, enables debugging across frontend/backend boundaries and monitoring service integration

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can complete registration (email, password, name) and receive valid JWT token within 3 seconds under normal network conditions
- **SC-002**: Authenticated users can perform full task CRUD operations with data persisted to database and visible across browser sessions
- **SC-003**: JWT validation completes in under 50ms per request when JWKS cache is warm (>95% cache hit rate after 10 requests)
- **SC-004**: Frontend handles 401 authentication errors by redirecting to login page within 1 second
- **SC-005**: All filter, search, sort, and pagination operations work correctly with real backend data, returning paginated results in under 2 seconds (50 tasks per page)
- **SC-006**: User isolation is enforced - cross-user access attempts return 403 Forbidden 100% of the time
- **SC-007**: JWKS cache reduces backend load by achieving >95% cache hit rate after initial warmup period (first 5 minutes)
- **SC-008**: System handles concurrent users without JWT validation becoming bottleneck (supports 100 concurrent users with p95 latency under 100ms)
- **SC-009**: Users experience zero authentication prompts during normal session (JWT token valid for minimum 1 hour)
- **SC-010**: Error messages are clear and actionable - users understand what went wrong and how to fix it (e.g., "Session expired, please log in again" vs generic "Error 401")
- **SC-011**: All authentication and authorization events are logged with correlation IDs enabling end-to-end request tracing from frontend to backend
- **SC-012**: Structured logging overhead adds less than 5ms latency to p95 request duration (measured via duration_ms field in logs)
- **SC-013**: Structured logging does not block request processing - console/file logging overhead does not cause user-facing failures
- **SC-014**: Correlation IDs are present in 100% of error responses and displayed to users for support troubleshooting
- **SC-015**: Zero sensitive data leaks in logs - automated log scanning detects no JWT tokens, passwords, or CSRF tokens in log output

### Assumptions

- Better Auth is already configured and running with JWKS endpoint accessible from backend
- Backend environment variable `BETTER_AUTH_JWKS_URL` points to valid JWKS endpoint
- Frontend and backend run on different domains/ports requiring CORS configuration
- JWT tokens have minimum 1-hour expiration time (Better Auth default)
- Backend has existing 14 REST API endpoints from Spec 003 that need JWT authentication added
- Backend API supports pagination via `?page=N&limit=M` query parameters with default page size of 50
- Frontend has existing UI components and mock data contexts from Spec 004 that need API integration
- EdDSA/Ed25519 is the signing algorithm used by Better Auth (not RS256 or HS256)
- User_id claim in JWT token matches database user IDs (string format)
- Better Auth `auth.getSession()` returns user object with `user.id` field after successful authentication
- Network conditions are stable (not testing offline mode or extreme latency scenarios)
- Browser supports modern JavaScript features (async/await, fetch API, localStorage)
