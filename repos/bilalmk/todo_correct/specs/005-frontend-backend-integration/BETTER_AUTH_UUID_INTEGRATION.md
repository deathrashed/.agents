# Better Auth UUID Integration - Implementation Guide

**Feature**: 005-frontend-backend-integration
**Date**: 2026-01-02
**Status**: ✅ Implemented and Tested

## Problem Solved

The application had **two conflicting authentication systems**:
1. Custom `users` table (UUID primary key) in FastAPI
2. Better Auth `user` table (String ID primary key) in frontend

This caused authentication failures because:
- Frontend registered users in Better Auth's `user` table
- Backend tried to find users in custom `users` table
- Users didn't exist in both tables → 401 Unauthorized errors

## Solution: Hybrid ID Approach

**Architecture**: Use Better Auth as single source of truth with UUID extension

```
Better Auth user table:
├── id: String (PK) - Better Auth manages this
└── uuid: UUID (UNIQUE, indexed) - Our application uses this
```

### Why Hybrid?
- **Better Auth requires String IDs** for internal operations (sessions, accounts)
- **Our application benefits from UUID** for type consistency and foreign keys
- **Best of both worlds**: Better Auth compatibility + application consistency

### 3. Frontend Changes

#### Better Auth Configuration (`frontend/src/lib/auth.ts`)
```typescript
export const auth = betterAuth({
  // ... existing config

  // ⭐ Hook to fetch UUID after user creation
  hooks: {
    user: {
      created: async ({ user }) => {
        // Fetch UUID generated by database
        const result = await pool.query(
          'SELECT uuid FROM "user" WHERE id = $1',
          [user.id]
        );
        const uuid = result.rows[0]?.uuid;

        // Return user with UUID for JWT inclusion
        return { ...user, uuid };
      }
    }
  },

  // ⭐ Include UUID in JWT payload
  plugins: [
    jwt({
      algorithm: "EdDSA",
      async jwt(user, session) {
        return {
          uuid: user.uuid,  // Custom claim
        };
      },
    }),
  ],
});
```

**Key Changes**:
- ✅ Added: `user.created` hook to fetch UUID
- ✅ Added: `jwt()` function to include UUID in token

## UUID Auto-Generation Flow

```
1. User registers via frontend
   └── authClient.signUp.email({ email, password, name })

2. Better Auth creates user in database
   └── INSERT INTO "user" (id, email, name, ..., uuid)
   └── uuid column has server_default=gen_random_uuid()
   └── PostgreSQL auto-generates UUID ⭐

3. Better Auth hook fires (user.created)
   └── SELECT uuid FROM "user" WHERE id = $1
   └── Fetches the auto-generated UUID
   └── Returns { ...user, uuid }

4. JWT token created with UUID
   └── { sub: "user_abc123", uuid: "a1b2c3d4-..." }
   └── Stored in httpOnly cookie

5. Backend extracts UUID from JWT
   └── payload.get("uuid")
   └── Queries User.where(uuid == ...)
```

**Important**: UUID is generated **automatically by the database**, not by application code. This ensures consistency and avoids race conditions.

## Common Issues & Solutions

### Issue: "JWT token missing 'uuid' claim"
**Solution**: Ensure Better Auth hook is configured correctly and JWT plugin includes `uuid` in payload.

### Issue: "User not found in database"
**Solution**: Check that migration 1 added UUID column and migration 3 dropped old `users` table.

### Issue: "Foreign key constraint violation"
**Solution**: Run migration 2 to update foreign keys to point to `user.uuid`.

### Issue: "BETTER_AUTH_SECRET mismatch"
**Solution**: Ensure `.env` files in both frontend and backend have identical `BETTER_AUTH_SECRET` values.
