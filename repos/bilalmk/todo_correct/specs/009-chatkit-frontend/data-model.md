# Data Model: ChatKit Frontend

**Feature**: 009-chatkit-frontend
**Phase**: Phase 1 - Data Model Design
**Date**: 2026-01-15

## Overview

This document defines the client-side state models, TypeScript interfaces, and React Context state for the ChatKit chatbot frontend integration. Unlike backend models (database schemas), these are ephemeral client-side structures.

---

## 1. Client-Side State Models

### 1.1 ChatBotSession

**Purpose**: Represents the current state of the chatbot popup.

**Lifecycle**: Created when popup opens, destroyed when popup closes.

**TypeScript Definition**:
```ts
// types/chatkit.d.ts
interface ChatBotSession {
  isOpen: boolean;
  threadId: string | null;          // UUID of current conversation thread
  isLoading: boolean;
  error: string | null;
  lastActivityTimestamp: string;    // ISO 8601
}
```

**State Management**: React `useState` hook in `ChatBotPopup.tsx`

**Example**:
```ts
const [session, setSession] = useState<ChatBotSession>({
  isOpen: false,
  threadId: null,
  isLoading: false,
  error: null,
  lastActivityTimestamp: new Date().toISOString(),
});
```

---

### 1.2 ChatMessage

**Purpose**: Represents a single message in the chat interface.

**Lifecycle**: Created when message sent/received, persists in UI until conversation cleared.

**TypeScript Definition**:
```ts
// types/chatkit.d.ts
interface ChatMessage {
  id: string;                       // UUID generated by ChatKit SDK
  role: 'user' | 'assistant' | 'system';
  content: string;                  // Message text
  timestamp: string;                // ISO 8601
  metadata?: {
    correlationId?: string;         // For tracing backend requests
    toolCalls?: ToolCall[];         // MCP tool invocations
  };
}

interface ToolCall {
  toolName: string;                 // 'add_task', 'list_tasks', etc.
  status: 'start' | 'success' | 'error';
  result?: any;
  error?: string;
}
```

**State Management**: Managed internally by `useChatKit` hook from OpenAI SDK

**Example**:
```ts
const { messages } = useChatKit({
  // ... config
});

// messages: ChatMessage[]
messages.map(msg => (
  <div key={msg.id} className={msg.role === 'user' ? 'text-right' : 'text-left'}>
    {msg.content}
  </div>
));
```

### 1.3 StreamingState

**Purpose**: Tracks the state of an in-progress AI response.

**Lifecycle**: Created when streaming starts, destroyed when streaming completes.

**TypeScript Definition**:
```ts
// types/chatkit.d.ts
interface StreamingState {
  isStreaming: boolean;
  partialContent: string;           // Accumulated text so far
  startTime: string;                // ISO 8601
  firstTokenTime: string | null;    // ISO 8601 (time to first token metric)
}
```

**State Management**: Internal to `useChatKit` hook (auto-managed)

**Example**:
```ts
const { isStreaming } = useChatKit({
  // ... config
});

{isStreaming && <div className="animate-pulse">AI is typing...</div>}
```

### 1.4 RequestContext

**Purpose**: Metadata injected into every chatbot API request.

**Lifecycle**: Created per request, sent to backend, discarded after response.

**TypeScript Definition**:
```ts
// types/chatkit.d.ts
interface RequestContext {
  userId: string;                   // UUID from JWT token
  correlationId: string;            // UUID for request tracing
  pageContext: {
    url: string;
    title: string;
    referrer: string;
  };
  timestamp: string;                // ISO 8601
}
```

**State Management**: Constructed in custom fetch interceptor

**Example**:
```ts
// lib/chatkit-config.ts
const customFetch = async (url: string, options: RequestInit) => {
  const session = await authClient.getSession();
  const userId = await getUserUuidFromSession();

  const context: RequestContext = {
    userId,
    correlationId: crypto.randomUUID(),
    pageContext: {
      url: window.location.href,
      title: document.title,
      referrer: document.referrer,
    },
    timestamp: new Date().toISOString(),
  };

  // Inject context into request body
  const body = JSON.parse(options.body as string);
  body.context = context;

  return fetch(url, {
    ...options,
    body: JSON.stringify(body),
  });
};
```

### 1.5 TaskEvent

**Purpose**: Custom event for real-time task synchronization between chatbot and dashboard.

**Lifecycle**: Emitted after task operation, consumed by dashboard, then garbage collected.

**TypeScript Definition**:
```ts
// lib/events/task-events.ts
export type TaskEventType =
  | 'task:created'
  | 'task:updated'
  | 'task:deleted'
  | 'task:completed';

export interface TaskEvent {
  type: TaskEventType;
  taskId?: number;                  // Optional (bulk operations may not have specific ID)
  timestamp: string;                // ISO 8601
  source: 'chatbot' | 'dashboard';  // Where event originated
}
```

**State Management**: Custom event via `window.dispatchEvent`

**Example**:
```ts
// Emit event (in TaskContext after API call)
export const emitTaskEvent = (event: TaskEvent) => {
  window.dispatchEvent(new CustomEvent('taskChange', { detail: event }));
};

// Listen for event (in Dashboard page)
export const onTaskEvent = (callback: (event: TaskEvent) => void) => {
  const handler = (e: Event) => callback((e as CustomEvent).detail);
  window.addEventListener('taskChange', handler);
  return () => window.removeEventListener('taskChange', handler);
};

// Usage
useEffect(() => {
  const unsubscribe = onTaskEvent((event) => {
    if (event.source === 'chatbot') {
      refreshTasks(); // Re-fetch from API
    }
  });

  return unsubscribe; // Cleanup
}, []);
```

## 2. React Context State Extensions

### 2.1 TaskContext Enhancements

**Purpose**: Add event emission capability to existing TaskContext.

**Changes**:
```ts
// contexts/TaskContext.tsx

// Existing interface (no changes)
interface TaskContextValue {
  tasks: Task[];
  isLoading: boolean;
  // ... existing methods
}

// NEW: Emit events after successful operations
const addTask = async (input: TaskCreate) => {
  const newTask = await apiClient.post(...);
  setTasks([newTask, ...tasks]);

  // Emit event for chatbot to listen
  emitTaskEvent({
    type: 'task:created',
    taskId: newTask.id,
    timestamp: new Date().toISOString(),
    source: 'dashboard',
  });

  return newTask;
};

const updateTask = async (id: number, updates: Partial<Task>) => {
  const updatedTask = await apiClient.patch(...);
  setTasks(tasks.map(t => t.id === id ? updatedTask : t));

  // Emit event
  emitTaskEvent({
    type: 'task:updated',
    taskId: id,
    timestamp: new Date().toISOString(),
    source: 'dashboard',
  });
};

// Similar for deleteTask, completeTask
```

**Rationale**: Allows dashboard and chatbot to notify each other of task changes without tight coupling.

## 3. Database Models (Backend - Reference Only)

The following models already exist in the backend and are NOT part of this frontend feature. Listed here for reference.

### 3.1 Conversation (Backend)

**Schema** (existing in `backend/src/models/conversation.py`):
```python
class Conversation(SQLModel, table=True):
    __tablename__ = "conversations"

    conversation_id: UUID = Field(default_factory=uuid4, primary_key=True)
    user_id: UUID = Field(foreign_key="users.uuid", index=True)
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    deleted_at: Optional[datetime] = Field(default=None)
    metadata: Optional[dict] = Field(default=None, sa_column=Column(JSON))

    # Relationship
    messages: List["Message"] = Relationship(back_populates="conversation")
```

**Usage**: Backend persists conversation history here; frontend does NOT directly access this model.

---

### 3.2 Message (Backend)

**Schema** (existing in `backend/src/models/message.py`):
```python
class Message(SQLModel, table=True):
    __tablename__ = "messages"

    message_id: UUID = Field(default_factory=uuid4, primary_key=True)
    conversation_id: UUID = Field(foreign_key="conversations.conversation_id", index=True)
    user_id: UUID = Field(foreign_key="users.uuid", index=True)
    role: str = Field(max_length=20)  # 'user', 'assistant', 'system'
    content: str = Field(max_length=100000)
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    deleted_at: Optional[datetime] = Field(default=None)
    metadata: Optional[dict] = Field(default=None, sa_column=Column(JSON))

    # Relationship
    conversation: Conversation = Relationship(back_populates="messages")
```

**Usage**: Backend stores each message here; frontend loads via ChatKit SDK (proxied through backend).

## 6. Data Flow Sequence

### 6.1 User Sends Chat Message

```
1. User types message in ChatInterface
2. ChatInterface calls sendMessage(text) from useChatKit hook
3. useChatKit hook constructs RequestContext (userId, correlationId, pageContext)
4. Custom fetch interceptor injects JWT token (if needed - actually from httpOnly cookie)
5. POST /api/chatkit (Next.js proxy route)
6. Next.js route extracts JWT from httpOnly cookie
7. Forwards to backend POST /api/chatkit/chat with Authorization: Bearer <JWT>
8. Backend validates JWT, creates/loads Conversation from database
9. Backend calls OpenAI Agents SDK with MCP tools
10. Backend streams SSE events back to Next.js proxy
11. Next.js proxy streams SSE to frontend
12. useChatKit hook parses SSE, updates messages array
13. React re-renders ChatInterface with new messages
```

### 6.2 Chatbot Creates Task

```
1. AI response includes tool call: add_task(user_id, title, description)
2. Backend MCP server executes add_task tool
3. Backend creates Task in database
4. Backend returns SSE event: tool.call.result
5. useChatKit hook receives event, updates UI
6. (Frontend does NOT emit TaskEvent yet - chatbot doesn't have direct access to TaskContext)
7. User sees confirmation message: "Task created successfully!"
8. User closes chatbot popup
9. Dashboard page's useEffect triggers refreshTasks() on chatbot close
10. Dashboard fetches updated task list from API
11. New task appears in dashboard
```

**Note**: Current design has chatbot and dashboard as separate contexts. Consider adding a global event emitter if real-time sync becomes a P0 requirement.

## 7. Validation Rules

### 7.1 Client-Side Validation

| Field | Rule | Error Message |
|-------|------|---------------|
| `ChatMessage.content` | Not empty, max 10,000 chars | "Message cannot be empty" / "Message too long (max 10,000 characters)" |
| `RequestContext.userId` | Valid UUID v4 format | "Invalid user ID format" |
| `TaskEvent.type` | One of allowed types | "Invalid task event type" |

### 7.2 State Invariants

| Invariant | Enforcement | Rationale |
|-----------|-------------|-----------|
| Only one active conversation per user | Backend enforces (database query) | Simplifies state management |
| threadId must exist before sending messages | Frontend checks in sendMessage | Prevents orphaned messages |
| StreamingState.isStreaming = true only during SSE | ChatKit SDK manages | Ensures UI consistency |

## Summary

**Client-Side State Models**:
1. `ChatBotSession` - Popup state (open/closed, loading, error)
2. `ChatMessage` - Individual messages in conversation
3. `StreamingState` - In-progress AI response tracking
4. `RequestContext` - Metadata for each API request
5. `TaskEvent` - Real-time sync events between chatbot and dashboard

**Backend Models (Reference)**:
1. `Conversation` - Database table for conversation persistence
2. `Message` - Database table for message history

**State Management**:
- React `useState` for component-local state
- `useChatKit` hook for ChatKit SDK integration
- Custom events for cross-component communication
- TaskContext for shared task state
